SECURITY REASONING FRAMEWORK:
Think like an attacker who has compromised one identity and wants to escalate privileges, move laterally, and reach high-value targets. Think like an IAM architect who must verify that every identity follows least-privilege, every trust relationship is intentional, and every permission boundary is enforced.

For EVERY identity and access-related resource you encounter (IAM users, roles, policies, groups, instance profiles, service-linked roles, access keys, etc.), systematically reason through these security dimensions:

1. LEAST PRIVILEGE — Does this identity have more permissions than it needs?
   Ask: Are there wildcard actions (e.g., s3:*, ec2:*, *)? Are resource constraints overly broad or missing? Are there unused but dangerous permissions? Could this identity access services it has no business using? Are there condition keys that meaningfully constrain access, or are they absent?
   Example: An IAM user policy with Action=["ec2:*", "s3:*", "lambda:*"] and Resource="*", a role that can iam:PassRole to any role, an instance profile with admin-equivalent permissions.

2. TRUST & DELEGATION — Are trust relationships properly scoped?
   Ask: Who can assume this role? Is the trust policy limited to specific accounts, services, or principals? Are there cross-account trust relationships without external ID or org boundary? Can roles be assumed without MFA? Are there circular or chained assume-role paths?
   Example: A role trust policy allowing Principal="*", cross-account trust without aws:ExternalId condition, service-linked roles with overly broad trust policies.

3. CREDENTIAL HYGIENE — Are identity credentials properly managed?
   Ask: Are IAM access keys rotated? Is MFA enforced for privileged operations? Are there console-accessible users without MFA? Are there long-lived static credentials where role-based access would be more appropriate? Are root account credentials properly secured?
   Example: An IAM user without MFA enabled, access keys created without rotation policy, console-enabled users with powerful policies but no MFA requirement.

4. PERMISSION BOUNDARIES — Are there guardrails limiting maximum permissions?
   Ask: Do privileged roles and users have permission boundaries? Are there SCP-equivalent controls? Can delegated admins create roles exceeding their own permissions? Are CI/CD identities bounded?
   Example: IAM roles without permissions_boundary, admin-creating roles that can bypass the creator's own limitations, Lambda execution roles without boundary constraints.

5. PRIVILEGE ESCALATION PATHS — Can an identity increase its own permissions?
   Ask: Can this identity create new roles, attach policies, or modify its own permissions? Can it pass roles to compute services? Can it access credentials of more privileged identities? Are there iam:CreateRole, iam:AttachRolePolicy, iam:PassRole, sts:AssumeRole chains?
   Example: A user who can iam:CreateRole and iam:AttachRolePolicy can create an admin role, a Lambda function that can iam:PassRole to any role can escalate through a new function.

6. BLAST RADIUS — If this identity is compromised, what's the damage?
   Ask: What data stores, compute resources, and management planes can this identity reach? Can it pivot to other identities? Can it modify security controls or disable logging? What's the worst case?
   Example: An EC2 instance profile that can access all S3 buckets, modify security groups, and assume other roles — compromising one instance compromises the entire environment.

IMPORTANT: The examples above are illustrative, not exhaustive. For every IAM resource, reason through ALL six dimensions. Examine every policy document, every trust relationship, every permission attachment. Think about what an attacker COULD do with each identity, not just what the identity was INTENDED to do. Do not limit your analysis to only the patterns listed above.
