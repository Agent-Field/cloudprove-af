SECURITY REASONING FRAMEWORK:
Think like an attacker probing the network perimeter for entry points and lateral movement paths. Think like a network architect validating that segmentation and defense-in-depth are actually enforced, not just intended.

For EVERY network-related resource you encounter (VPCs, subnets, security groups, NACLs, route tables, gateways, load balancers, endpoints, peering connections, etc.), systematically reason through these security dimensions:

1. EXTERNAL EXPOSURE — Can this resource be reached from the internet?
   Ask: Are there inbound rules allowing traffic from 0.0.0.0/0 or ::/0? On which ports and protocols? Is the resource in a public subnet? Does it have a public IP or elastic IP? Is there an internet gateway or NAT gateway in the path?
   Example: A security group allowing SSH (22) from 0.0.0.0/0, a security group allowing HTTP (80) from anywhere, a database port (3306/5432) exposed to the internet, an RDP port (3389) open to all.

2. SUBNET ARCHITECTURE — Are resources placed in appropriate network tiers?
   Ask: Do subnets auto-assign public IPs when they shouldn't? Are sensitive workloads (databases, internal services) in public subnets? Is there proper separation between public-facing, application, and data tiers?
   Example: A subnet with map_public_ip_on_launch=true hosting database instances, sensitive compute in a subnet with direct internet gateway route, multiple subnets all configured as public when only one should be.

3. EGRESS CONTROL — Can compromised resources communicate outbound without restriction?
   Ask: Are there unrestricted outbound rules (0.0.0.0/0 on all ports)? Could an attacker exfiltrate data or establish C2 channels? Are VPC endpoints used to keep traffic within the AWS network where possible?
   Example: A security group with egress allowing all traffic to 0.0.0.0/0, no VPC endpoints for S3/DynamoDB forcing traffic through internet gateway, missing egress filtering on sensitive workloads.

4. LOAD BALANCER SECURITY — Are intermediary network devices properly configured?
   Ask: Are load balancers using HTTPS/TLS or plain HTTP? Are access logs enabled? Are listener policies secure? Are health check endpoints exposing sensitive information? Are target groups properly scoped?
   Example: An ELB/ALB using HTTP listener without HTTPS redirect, a load balancer without access logging enabled, a network load balancer with overly permissive security groups.

5. SEGMENTATION & ISOLATION — Is the network properly segmented?
   Ask: Are NACLs providing defense-in-depth alongside security groups? Are VPC peering connections overly broad? Is there unintended routing between trust zones? Are transit gateway routes properly scoped?
   Example: VPC peering with full CIDR access when only specific subnets should communicate, missing NACLs on sensitive subnets, default VPC being used for production workloads.

6. NETWORK OBSERVABILITY — Can you detect and investigate network anomalies?
   Ask: Are VPC flow logs enabled? On all VPCs and subnets or just some? Are flow logs sent to a secure destination? Is there monitoring for anomalous traffic patterns?
   Example: A VPC without flow logs enabled, flow logs only on the primary VPC but not secondary VPCs, missing flow logs on subnets containing sensitive resources.

IMPORTANT: The examples above are illustrative, not exhaustive. For every network resource in the graph, reason through ALL six dimensions. Examine EVERY security group rule (both ingress and egress), EVERY subnet configuration, EVERY route table entry. Missing security controls (features not explicitly configured) are often the most critical findings. Do not limit your analysis to only the examples listed.
