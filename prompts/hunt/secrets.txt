SECURITY REASONING FRAMEWORK:
Think like an attacker who has gained read access to this repository or one of its deployment artifacts. Every credential, key, password, token, or sensitive configuration value is a potential pivot point to deeper compromise. Think like a secrets management architect who must ensure no credential can be extracted, reused, or escalated beyond its intended scope.

For EVERY resource in the graph, systematically reason through these security dimensions:

1. CREDENTIAL EXPOSURE — Are secrets visible in plaintext anywhere in the IaC?
   Ask: Are there hardcoded access keys, secret keys, passwords, tokens, or API keys? Are secrets in environment variables, user_data scripts, variable defaults, local values, or outputs? Could someone with repo access extract working credentials?
   Example: AWS access keys in Lambda environment variables, database passwords in Terraform variable defaults, API tokens in EC2 user_data bootstrap scripts, secret values exposed through Terraform outputs.

2. SECRET LIFECYCLE — Are secrets managed with proper rotation and governance?
   Ask: Are access keys rotated? Are KMS keys set for automatic rotation? Do Secrets Manager secrets have rotation configured? Are there stale credentials that should have been decommissioned? Is there a key management strategy?
   Example: IAM access keys without rotation policy, KMS keys without enable_key_rotation=true, Secrets Manager secrets without rotation_lambda_arn, long-lived static credentials instead of short-lived role-based access.

3. SECRET DISTRIBUTION — How do secrets flow through the infrastructure?
   Ask: Are secrets injected at build time vs runtime? Are they passed through secure channels (Secrets Manager, SSM SecureString) or insecure ones (environment variables, user_data, plaintext config)? Could secrets leak through logs, outputs, or intermediate storage?
   Example: Database credentials hardcoded in user_data instead of retrieved from Secrets Manager at runtime, SSM Parameters using String type instead of SecureString for sensitive values, secrets passed through Terraform outputs that appear in state files.

4. BLAST RADIUS — What can an attacker do with a compromised credential?
   Ask: If this secret is compromised, what resources become accessible? Does it grant access to other secrets or credentials? Can it be used to escalate privileges? What data stores, compute resources, or management planes become reachable?
   Example: An IAM access key for a user with wildcard permissions grants access to every AWS service, a database password exposed in user_data gives direct access to an unencrypted RDS instance.

5. PROTECTION DEPTH — Are there layers of defense around secret material?
   Ask: Is the secret encrypted? With what key? Who has access to that key? Are there access policies limiting who can retrieve the secret? Is there monitoring for secret access? Could the encryption be bypassed?
   Example: Secrets encrypted with AWS-managed keys instead of CMKs (less control), KMS key policies allowing broad decrypt access, missing CloudTrail data events for Secrets Manager access.
