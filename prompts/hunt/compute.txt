SECURITY REASONING FRAMEWORK:
Think like an attacker who has gained code execution on one compute resource and wants to escalate, persist, move laterally, or exfiltrate data. Think like a platform security architect who must ensure every workload is hardened, isolated, and observable by default.

For EVERY compute-related resource you encounter (EC2 instances, Lambda functions, ECS tasks/services, EKS clusters/node groups, auto-scaling groups, launch templates, container registries, batch jobs, etc.), systematically reason through these security dimensions:

1. WORKLOAD HARDENING — Is this compute resource configured securely?
   Ask: Is IMDSv2 enforced (http_tokens=required)? Are root volumes encrypted? Is the OS/runtime patched and current? Are bootstrap scripts secure? Are there dangerous capabilities or privilege escalations enabled?
   Example: An EC2 instance without IMDSv2 enforcement allowing SSRF-based credential theft, a launch template using a deprecated AMI, user_data scripts downloading packages over HTTP, an ECS task definition with privileged=true.

2. CONTAINER & ORCHESTRATION SECURITY — Are containerized workloads properly configured?
   Ask: Are container images from trusted registries? Are image tags immutable? Is image scanning enabled? Are pod security contexts restrictive? Is the orchestration control plane hardened? Are secrets properly injected (not baked into images)?
   Example: An ECR repository with image_tag_mutability="MUTABLE" allowing supply chain attacks, an ECR repository without image scanning enabled (image_scanning_configuration not set or scan_on_push=false), an EKS cluster with endpoint_public_access=true exposing the API server, EKS without enabled_cluster_log_types (no audit logging), EKS without encryption_config for secrets at rest.

3. STORAGE ATTACHED TO COMPUTE — Are data volumes properly protected?
   Ask: Are EBS volumes encrypted? Are EBS snapshots encrypted? Are ephemeral volumes used for sensitive data? Are volume attachments to appropriate instances? Could snapshot sharing expose data?
   Example: An EBS volume with encrypted=false, an EBS snapshot without encryption that could be shared publicly, a volume attached to a public-facing instance containing sensitive data.

4. EXECUTION IDENTITY — Does the workload's identity follow least privilege?
   Ask: What role or instance profile is attached? Does it have excessive permissions? Can it access resources beyond its workload scope? Can it assume other roles or pass roles? Is there a permission boundary?
   Example: A Lambda function with an execution role granting s3:* and ec2:*, an EC2 instance profile that can assume any role, a workload identity without a permission boundary.

5. NETWORK POSITION — Is the compute resource properly isolated?
   Ask: Is it in a public subnet? Does it have a public IP? Are security groups restrictive? Can it reach sensitive internal resources? Is it exposed to the internet through a load balancer?
   Example: An EC2 instance with associate_public_ip_address=true in a subnet that doesn't need internet-facing instances, a compute resource with security groups allowing all inbound traffic.

6. OBSERVABILITY & MANAGEMENT — Can you detect and respond to incidents on this workload?
   Ask: Is SSM management enabled for patching and incident response? Are logs forwarded? Is monitoring configured? Are there health checks? Can you audit what the workload is doing?
   Example: EC2 instances without SSM agent or management role, Lambda functions without log group retention policies, EKS clusters without audit logging enabled.

IMPORTANT: The examples above are illustrative, not exhaustive. For every compute resource in the graph, reason through ALL six dimensions. Examine EVERY attribute — missing security features (attributes not explicitly set) are often the most critical findings. A single compute resource may have findings across multiple dimensions. Do not limit your analysis to only the patterns listed above.
