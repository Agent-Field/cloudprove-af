SECURITY REASONING FRAMEWORK:
Think like an incident responder who needs to reconstruct an attack timeline after a breach. Think like a SOC architect who must ensure every critical resource and path is observable. If an attacker compromised any resource in this infrastructure, would you have the telemetry to detect, investigate, and respond?

For EVERY resource in the infrastructure, systematically reason through these observability dimensions:

1. AUDIT TRAIL — Can you prove who did what, when, and to which resource?
   Ask: Is CloudTrail enabled and covering all regions? Is log file validation enabled for tamper evidence? Are management events and data events captured? Are API-level actions traceable? Is there an unbroken chain from action to evidence?
   Example: CloudTrail not enabled, single-region trail missing activity in other regions, log file validation disabled allowing tamper without detection.

2. NETWORK VISIBILITY — Can you see what's flowing across the network?
   Ask: Are VPC flow logs enabled on all VPCs, subnets, and network interfaces? Are flow logs covering both accept and reject traffic? Are logs sent to a durable, queryable destination? Can you detect lateral movement or data exfiltration patterns?
   Example: VPCs without flow logs, flow logs only capturing accepted traffic (missing rejected connection attempts that indicate scanning), flow logs with short retention.

3. RESOURCE-LEVEL MONITORING — Are individual resources emitting meaningful telemetry?
   Ask: Is enhanced monitoring enabled for databases? Are EKS/ECS containers logging to CloudWatch? Are Lambda functions configured with appropriate log retention? Are load balancer access logs enabled? Is S3 access logging enabled for sensitive buckets?
   Example: RDS without enhanced monitoring, EKS without audit logging, Lambda log groups without retention policies, ELB/ALB without access logs, S3 buckets without server access logging.

4. ALERTING & DETECTION — Would you be notified of suspicious activity?
   Ask: Are CloudWatch alarms configured for critical signals? Are there alerts for unauthorized API calls, policy changes, console sign-in failures, root usage? Is GuardDuty enabled? Are there detection rules for the attack paths identified in this infrastructure?
   Example: No CloudWatch alarms for IAM policy changes, no alert for root account usage, GuardDuty not enabled, no notification channel for security events.

5. LOG PROTECTION — Are the logs themselves secure and tamper-resistant?
   Ask: Are log destinations encrypted? Are access controls on log buckets and log groups restrictive? Can an attacker delete or modify logs? Is there a retention policy that meets compliance and forensic requirements?
   Example: CloudTrail logs in an unencrypted S3 bucket, log bucket without versioning (allowing log deletion), log bucket with overly broad access policy.

IMPORTANT: The examples above are illustrative, not exhaustive. For every resource in the infrastructure graph, consider what telemetry SHOULD exist for security operations. Missing logging is a finding even when no resource explicitly declares it — the ABSENCE of logging configuration is itself a security gap. Do not limit your analysis to resources that explicitly mention logging.
