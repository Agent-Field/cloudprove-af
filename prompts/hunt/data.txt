SECURITY REASONING FRAMEWORK:
Think like an attacker who has found one foothold and wants to reach, read, corrupt, or destroy data. Think like a compliance auditor who must certify every data store meets security baselines. Your job is to find every way data can be exposed, lost, corrupted, or accessed by unauthorized parties.

For EVERY data-related resource you encounter (any storage, database, queue, cache, stream, or key management resource), systematically reason through these security dimensions:

1. CONFIDENTIALITY — Can unauthorized parties read this data?
   Ask: Is encryption at rest enabled? Is encryption in transit enforced? Are access policies overly broad or public? Are there cross-account sharing risks? Can the data be accessed through unintended paths?
   Example: An S3 bucket without server-side encryption, an RDS instance without storage_encrypted=true, an Elasticsearch domain without node-to-node encryption, a Neptune cluster without storage encryption.

2. INTEGRITY — Can this data be tampered with or its provenance lost?
   Ask: Is versioning enabled? Are there immutability controls? Can objects be overwritten without audit trail? Are there lifecycle policies that could silently delete data? Can tags be mutated to bypass policies?
   Example: An S3 bucket without versioning means ransomware can overwrite objects irreversibly. A container registry with mutable tags means image provenance is unverifiable.

3. AVAILABILITY — Can this data be lost, and can the system recover?
   Ask: Are backups configured? Is there retention policy? Is multi-AZ or replication set up for critical stores? Are snapshots encrypted? Is there a disaster recovery path?
   Example: An RDS instance with backup_retention_period=0, an EBS volume without snapshots, a DynamoDB table without point-in-time recovery.

4. ACCESS CONTROL — Who can reach this data and through what paths?
   Ask: Are bucket policies, resource policies, or ACLs overly permissive? Is public access blocked at the account or resource level? Are there conditions limiting access? Do IAM paths grant broader access than intended?
   Example: An S3 bucket ACL allowing public-read, an Elasticsearch domain policy with Principal="*", a database publicly accessible from the internet.

5. OBSERVABILITY — Would you know if this data was accessed or exfiltrated?
   Ask: Is access logging enabled? Are there audit trails? Is object-level logging configured for sensitive buckets? Are CloudWatch alarms monitoring abnormal access patterns?
   Example: An S3 bucket without access logging, a database without enhanced monitoring, missing CloudTrail data events for sensitive buckets.

6. KEY MANAGEMENT — Are encryption keys properly governed?
   Ask: Is key rotation enabled? Are key policies overly permissive? Are grants scoped narrowly? Could key compromise cascade to multiple data stores?
   Example: A KMS key without automatic rotation, key policies allowing broad decrypt access, secrets encrypted with AWS-managed keys instead of CMKs for sensitive workloads.

IMPORTANT: The examples above are illustrative, not exhaustive. For every data resource in the graph, reason through ALL six dimensions. A single resource may have findings across multiple dimensions. Do not limit your analysis to only the examples listed — discover issues by reasoning about what SHOULD be configured for each resource based on cloud security best practices.
